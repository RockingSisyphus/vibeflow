# 训练性能导向内核改进计划

## 最终内核改进目标

把 VibeTrain 从：

```text
运行时解释执行 + 每个 node 边界强校验 + 数据内容可 snapshot
```

改成：

```text
编译期检查流程结构 + 运行期按编译产物高速执行 + node 间任意 Python 对象直通
```

核心定位：

```text
VibeTrain 审计流程，不审计数据内容。
```

也就是说，VibeTrain 负责回答：

```text
有哪些 node？
node 怎么连接？
控制流怎么走？
哪些步骤有副作用？
训练目前运行到哪个阶段？
失败发生在哪个 node？
```

不负责回答：

```text
a -> b 之间具体传了什么 Python 对象？
tensor shape 是什么？
model 内部参数是什么？
optimizer state 具体长什么样？
```

---

## 改进 1：取消 node 间数据类型限制

当前问题：

```text
PyTorch 训练需要在 node 之间传 tensor、model、optimizer、loss、autograd graph 等 Python 对象。
如果内核要求输出必须 JSON-safe / 可 snapshot，就会天然排斥这类对象。
```

修改方向：

```text
runtime context 允许保存任意 Python object。
node 输出直接按引用写入 context。
node 输入直接从 context 按引用读取。
不再要求输出值 JSON serializable。
不再默认 snapshot 输出值。
```

保留的检查：

```text
requires/provides 的 key 是否存在。
node 是否声明自己需要/提供哪些 key。
下游需要的 key 是否能由上游流程提供。
```

不再检查：

```text
value 类型
value schema
value 是否 JSON 化
value 是否可深拷贝
value 内容
```

为什么要这么改：

```text
VibeTrain 的目标是流程审计，不是数据内容审计。
PyTorch 的核心对象天然不是 JSON 数据。
强制 JSON 化会导致性能差、实现绕、opaque 到处出现。
```

为什么能解决问题：

```text
tensor、model、optimizer、autograd graph 可以在 node 间直接按引用传递。
forward -> loss -> backward -> optimizer_step 这种细粒度训练图可以自然成立。
不需要 external 黑箱，也不需要 artifact 化每个中间对象。
```

---

## 改进 2：trace 不记录数据内容，只记录流程事件

当前问题：

```text
如果 trace 默认记录 node 输出内容，就会遇到 tensor/model 无法记录、记录成本高、日志巨大、训练被拖慢。
```

修改方向：

```text
trace 默认只记录：
  node_id
  start/end
  duration
  input keys
  output keys
  output type name 可选
  exception
```

不记录：

```text
真实 tensor 内容
model 参数
optimizer state
完整 Python object repr
```

正式训练只维护：

```text
runtime.current_node
block/node 边界事件
错误定位信息
```

debug 模式再开启更详细 trace。

为什么要这么改：

```text
训练时关心“运行到哪一步、哪里失败、耗时多少”，不需要每条边的数据内容。
数据内容 trace 对 PyTorch 对象成本极高，且多数时候没价值。
```

为什么能解决问题：

```text
不会因为 trace tensor/model 导致序列化、同步、日志爆炸。
仍然能定位当前 node 和失败位置。
流程审计保留，数据内容审计去掉。
```

---

## 改进 3：预编译 graph/nodeset 为执行计划

当前问题：

解释型 runtime 每执行一个 node 都可能重复做：

```text
查图
查 edge
解析 requires/provides
组装 inputs
查找 node callable
查找 plugin hook
判断下一个 node
记录 trace
```

这些在训练 step 高频执行时会被放大。

修改方向：

编译阶段生成 `ExecutionPlan`：

```text
NodeFrame:
  node_id
  callable
  input_keys
  output_keys
  params
  next frame
  decision route info
  hook calls
```

运行时直接执行：

```python
for frame in plan:
    inputs = pick(context, frame.input_keys)
    outputs = frame.callable(inputs, frame.params)
    context.update(outputs)
```

编译期提前完成：

```text
拓扑排序
可达性检查
decision edge 结构检查
requires/provides key 检查
input/output key 映射
node callable 绑定
plugin hook 绑定
nodeset 展开/嵌套关系
```

为什么要这么改：

```text
训练热路径不能每 step 都重复解释流程图。
这些结构性信息在 run 开始前已经确定，不应该运行时反复算。
```

为什么能解决问题：

```text
运行时从“动态解释 graph”变成“执行预计算好的计划”。
减少图查询、key 映射、hook 查找、edge 解析等通用调度成本。
```

---

## 改进 4：把 graph/nodeset 编译成 compiled block

当前问题：

即使有 `ExecutionPlan`，如果每个 node 仍然逐个回到通用调度器，仍然有边界成本。

修改方向：

把一个 pipeline 或 nodeset 编译成一个 Python execution block。

线性图：

```text
A -> B -> C
```

编译成：

```python
def compiled_block(ctx):
    runtime.current_node = "A"
    ctx.update(A(...))

    runtime.current_node = "B"
    ctx.update(B(...))

    runtime.current_node = "C"
    ctx.update(C(...))
```

decision/cycle 图：

```text
train_epoch -> should_continue?
yes -> train_epoch
no -> report
```

编译成：

```python
while True:
    train_epoch(...)
    route = should_continue(...)

    if route == "yes":
        continue
    if route == "no":
        break
```

为什么要这么改：

```text
普通 Python 拆成多个函数不慢，是因为只是函数调用。
VibeTrain node 慢，是因为每个 node 有 workflow 调度边界。
compiled block 可以把 workflow 调度从每个 node 降到 block 编译阶段。
```

为什么能解决问题：

```text
运行时更接近手写 PyTorch training loop。
node 仍然在流程图里可见，但执行时不需要每次走完整解释器。
```

## 改进 5：支持 trace policy

当前问题：

不同场景需要不同 trace 粒度：

```text
debug 需要完整 node trace。
正式训练需要低开销。
线上长跑只需要错误定位和 summary。
```

修改方向：

增加 trace 策略：

```text
full:
  每个 node start/end 都记录

boundary:
  只记录 block/nodeset 边界

sampled:
  每 N step 记录一次

off:
  不记录 trace，只维护 current_node 和异常上下文
```

为什么要这么改：

```text
训练框架不能默认逐 step 记录所有 node。
trace 是观测能力，不应该固定成为性能瓶颈。
```

为什么能解决问题：

```text
debug 时保留可解释性。
正式训练时降低 trace 开销。
出错时仍可通过 current_node 知道失败位置。
```

---

## 改进 6：plugin hook 编译期插桩，运行期按策略启用

当前问题：

如果每个 node 运行时都动态查找并调用 plugin hook，会增加调度开销。

修改方向：

编译时决定 hook 插入点：

```python
before_forward(ctx)
forward(ctx)
after_forward(ctx)
```

或者在正式训练中只插入 block 级 hook：

```python
before_train_step_block(ctx)
compiled_train_step(ctx)
after_train_step_block(ctx)
```

支持 hook 粒度：

```text
run-level
nodeset-level
block-level
node-level
```

正式训练默认：

```text
run-level + block-level
```

debug 才开：

```text
node-level
```

为什么要这么改：

```text
plugin 是观测和扩展点，但高频逐 node hook 会拖慢训练。
hook 是否存在、插在哪里，编译期就能决定。
```

为什么能解决问题：

```text
省掉运行时 hook 查找成本。
正式训练避免每个微节点都调用 plugin。
需要 debug 时仍可恢复完整 hook 能力。
```

## 改进 7：异步执行模型

当前问题：

程序流程图基本是顺序执行，但训练场景里经常需要旁路任务或并行任务：

```text
metrics / logging / diagnostics 需要后台写出。
模型多分支可能需要并行处理数据。
某些 CPU 后处理不应该阻塞 GPU 主训练路径。
```

同时，VibeTrain 改进后不再检查 node 之间具体传递的数据内容，所以核心不应该尝试自动合并异步分支的 context。

核心原则：

```text
VibeTrain 管异步控制流，不管异步数据合并。
```

修改方向：支持三种异步模式。

### Detached Async

用于 metrics / logging / diagnostics。

```text
启动后主流程不等。
不回写主 context。
失败默认 log_and_continue。
结束时可 flush_timeout。
```

适合：

```text
写 JSONL、TensorBoard、监控上报、诊断统计。
```

### Async Result

用于异步计算结果要回到主流程。

```text
异步分支返回一个 result。
result 写入显式 result_key。
下游 node 如果 requires 这个 key，就自动 join。
```

例子：

```text
b -> async(e -> f) provides f_result
c requires f_result
```

语义：

```text
c 执行前等待 f_result。
核心只做 future.result() -> context["f_result"]。
不自动 merge 整个 context。
```

### Shared Mutable Object

用于高级场景。

```text
主流程和异步分支共享同一个 Python 对象引用。
核心不管对象内部 mutation。
线程安全、锁、队列、actor 由业务代码自己负责。
```

适合：

```text
thread-safe metrics writer、cache、actor、data_store、训练状态管理器。
```

核心规则：

```text
不自动 merge async context。
异步分支要么 detached，要么通过 result_key 显式 join。
是否等待由 requires/provides 决定。
失败策略可配置：fail_on_join / fail_fast / log_and_continue。
result_key 冲突默认 overwrite_with_warning 或 error，二选一。
```

建议默认值：

```text
detached: log_and_continue + flush_timeout
joined: fail_on_join
result_key 冲突: overwrite_with_warning
```

为什么要这么改：

```text
自动 merge 任意 context 没有通用正确答案。
核心不知道异步分支到底改了什么，也不审计数据内容。
异步应该表达控制流，而不是把线程、锁、merge 规则暴露成核心复杂度。
```

为什么能解决问题：

```text
metrics 等旁路任务可以 detached，不阻塞主训练。
真正需要回主流程的并行计算通过 result_key 显式 join。
高级共享状态由业务对象自行保证线程安全，核心保持简单和宽容。
```

## 改进 8：保留 key 级契约和流程健康检查

虽然放开数据内容，但不要取消所有检查。

保留：

```text
node 是否声明 NODE_INFO / CONTRACT
requires/provides key 是否合理
pipeline start/end 是否存在
node 是否可达
cycle 是否经过 decision
io/data_store/document/process/decision 类型是否表达清楚
planned/implemented 规则
plugin health finding
```

取消：

```text
edge value 类型检查
JSON serializable 检查
output value snapshot 强制要求
普通运行时 value schema 检查
```

为什么要这么改：

```text
VibeTrain 的价值是流程结构治理，不是深度检查数据。
如果把流程检查也去掉，VibeTrain 就退化成普通脚本。
```

为什么能解决问题：

```text
保留流程图、控制流、节点职责、可达性、失败定位。
同时不阻碍 PyTorch 对象在流程中自然流动。
```

## 最终形态

理想训练执行模型：

```text
开发视角：
  load_batch -> forward -> loss -> backward -> optimizer_step -> log_metrics

编译视角：
  检查 node/key/edge/decision/cycle/plugin

运行视角：
  compiled_train_step(ctx)
  tensor/model/optimizer 按引用传递
  trace 只记边界/current_node
  metrics 异步写出
```

## 一句话总结

最终内核改进计划是：

```text
放开 node 间数据内容，
保留 key 级流程契约；
把检查前移到编译期，
把运行改成 compiled block；
trace 和 plugin 从默认逐 node 改为可配置插桩；
异步执行从自动 merge 改为 detached / result_key join / shared object 三种简单模式。
```

为什么这么改：

```text
因为 VibeTrain 要审计流程，不需要审计每条边上的对象内容；
PyTorch 训练需要任意对象按引用流动；
训练热路径不能每 step 承担解释型 workflow 调度和同步日志成本。
```

为什么能解决问题：

```text
任意 Python object 直通解决 PyTorch 跨 node 问题；
编译执行解决 node 边界调度问题；
轻量 trace 解决 snapshot/log 爆炸问题；
异步执行模型解决旁路记录和并行分支问题；
保留 key/graph 检查则继续维持 VibeTrain 的流程图价值。
```

## 当前项目现状核对

本节记录对当前 `vibeflow` 代码的核对结果。结论是：训练性能问题不是假设问题，当前实现确实把普通流程治理中的“数据可审计”成本带进了训练热路径。

### 已经符合目标的部分

```text
编译期已经具备流程结构检查：显式 edge、provider/consumer、cycle/decision、flow_kind。
运行产物 input_summary/output_summary 已经只写摘要，不写真实值。
runtime_trace.jsonl 当前记录 event 和 summary，没有直接写完整 outputs。
summaries.py 只记录 type/size/keys，不记录真实对象内容。
```

### 与训练目标冲突的部分

```text
runtime.py 每个 node 输入会 deepcopy(context.get(key))。
runtime.py 会 deepcopy(inputs) 后用 inputs != before 检查输入变异。
runtime.py 每个输出会调用 assert_runtime_output_snapshot。
runtime_validation.py 默认要求输出 json.dumps 成功。
context.py 初始化和 to_dict() 都会 deepcopy 整个 context。
runtime.py 的 before_run/after_run/decision 条件求值会触发 context.to_dict()。
runtime.py 每个 node 都动态查 registry、实例化 node、合并 config、查 plugin hook。
nodeset 当前会重新创建 PipelineRuntime，内部再次 compile 和解释执行。
plugin 当前只有逐 node/nodeset hook，没有 trace/hook 粒度策略。
purity_types.py 禁止 asyncio/threading/multiprocessing，当前没有训练旁路异步能力。
```

### 当前测试中固定下来的旧行为

```text
tests/unit/test/strict_mermaid_cli.py 明确测试“非 JSON snapshot 输出应失败”。
tests/unit/test/strict_runtime.py 明确测试运行时 JSON snapshot 失败会进入 trace。
这些测试需要随第一阶段一起改成“任意 Python object 可按引用直通”。
```

## 分阶段落地路线

### 第一阶段：对象按引用直通，取消运行时默认数据内容审计

目标：先让 PyTorch tensor/model/optimizer/loss/autograd graph 可以自然跨 node 传递。

修改范围：

```text
Context 默认按引用保存对象。
Context.to_dict() 不再深拷贝训练对象，或仅作为浅拷贝视图使用。
PipelineRuntime._run_pure_node() 直接按 key 取输入引用。
删除运行时 deepcopy 输入变异检查。
删除运行时默认 JSON snapshot 输出限制。
继续检查 outputs 必须是 Mapping。
继续检查输出 key 不能多、不能少。
继续保留 requires/provides key、flow edge、decision/cycle、planned/implemented 检查。
```

不做：

```text
不做 ExecutionPlan。
不做 compiled block。
不做 async。
不新增复杂策略系统。
```

验收标准：

```text
非 JSON 输出对象可以写入 context 并被下游按同一对象引用读取。
不可 deepcopy 的对象不会导致 runtime 失败。
runtime_trace/input_summary/output_summary 不记录真实对象内容。
旧的 key 合约错误仍然失败。
现有核心测试更新后通过。
```

### 第二阶段：RuntimeOptions 和 TracePolicy

目标：让 trace 和 plugin hook 不再成为训练热路径固定成本。

建议最小接口：

```python
RuntimeOptions(
    trace="full" | "boundary" | "off",
    node_hooks=True | False,
)
```

语义：

```text
full：保留当前逐 node/nodeset event。
boundary：只记录 run/nodeset/block 边界和失败。
off：只维护 current_node、stop_reason、异常上下文。
node_hooks=False：跳过 before_node/after_node，只保留 run-level hook。
```

### 第三阶段：ExecutionPlan

目标：把动态解释成本前移，但先不生成 Python 代码。

建议新增 `execution_plan.py`：

```python
@dataclass(frozen=True)
class NodeFrame:
    name: str
    node_type: str
    node: object
    requires: tuple[str, ...]
    provides: tuple[str, ...]
    params: Mapping[str, object]
    outgoing: tuple[EdgeSpec, ...]
    is_terminal: bool
    is_nodeset: bool
```

编译期提前完成：

```text
registry lookup。
node 实例化或 callable 绑定。
config merge。
incoming/outgoing edge 表。
flow_kind 绑定。
nodeset 子计划递归编译。
plugin hook 列表按 RuntimeOptions 绑定。
```

### 第四阶段：nodeset 子计划复用

目标：避免 nodeset 每次重新创建 `PipelineRuntime` 和重新 compile。

修改方向：

```text
ExecutionPlan 中为 nodeset frame 保存子 plan。
nodeset 执行直接调用子 plan runner。
nodeset 输入按引用传入。
nodeset exports 只按 key 写回父 context。
trace policy 决定是否记录内部节点。
```

### 第五阶段：可选 compiled block

目标：仅在线性高频训练图上进一步降低调度开销。

先支持：

```text
线性 block：无 decision/cycle 的连续节点序列。
简单 decision loop block：只支持当前 when 表达式可覆盖的 route。
```

暂不支持：

```text
任意 Python 代码生成。
任意复杂 CFG 优化。
自动并行。
自动 context merge。
```

### 第六阶段：异步能力

目标：支持 metrics/logging/diagnostics 旁路任务，不把核心变成并发框架。

先做两种：

```text
detached：启动后主流程不等，结束时 flush，失败默认记录 warning。
result_key：future 结果写入单个显式 key，下游 requires 这个 key 时 join。
```

仍然不做自动 merge context。共享对象直接按引用传，线程安全由业务对象负责。

## 后续修正计划：对照原始意图的缺口

阶段一到阶段六已经把训练性能方向的主要运行时阻塞点拆开并落地，但当前实现仍有几处只达到“第一版最小能力”，没有完全达到本文件前半部分的最终意图。

### 当前确认的缺口

```text
高：CLI 和分发包 run.py 不能传 RuntimeOptions。
    run_checked(..., runtime_options=...) 已支持，但普通命令行入口仍只能使用默认 full trace、node_hooks=True、execution="plan"。

高：async 仍是单 node call-site 能力，不是子流程/控制流级能力。
    当前 async 可以做 metrics/logging 单节点旁路，但 async(nodeset / e -> f) 还不能表达；runtime 还禁止 async nodeset。

中：result_key 必须在 provides 中的合约检查仍主要发生在 runtime。
    schema/parse 阶段只检查 result_key 是否存在，未把 result_key in provides 前移到配置/健康检查。

中：detached async flush 没有 timeout。
    后台 writer 卡死时，主流程结束后仍可能在 future.result() 或 executor shutdown(wait=True) 卡住。

中：execution="block" 目前是保守 block runner，不是最终 compiled block。
    它减少 ready queue 调度，但仍逐 node 走 _run_node、summary、hook、edge activation 和 context 查询。

中：CONTRACT.examples 仍默认要求实际输出 JSON snapshot。
    runtime 已不审计对象内容，但静态 example 对训练对象/普通 Python object 仍会被 JSON 校验挡住。

低：plugin hook 只是预绑定 runtime plugin 列表，不是 hook callable 插桩。
    每次 hook 调用仍会 getattr 查方法；node_hooks=False 能跳过 node hook，但未达到编译期 hook table。
```

这些缺口不推翻已完成阶段；它们说明当前实现是训练性能路线的第一轮闭环，而不是最终性能形态。

### 修正 1：补齐 CLI / 分发包 run.py 的低开销入口

目标：正式训练不必写 Python wrapper 才能启用低开销运行选项。

修改方向：

```text
src/vibeflow/cli.py run 增加：
  --trace full|boundary|off
  --execution plan|block
  --node-hooks / --no-node-hooks
  --async-flush-timeout <seconds>

distribution/kernel_development_pack/project_template/run.py run 增加同样参数。
```

建议增加 preset：

```text
--runtime-profile debug:
  trace="full"
  node_hooks=True
  execution="plan"

--runtime-profile train:
  trace="boundary"
  node_hooks=False
  execution="plan"
```

`execution="block"` 不建议作为 CLI 默认值，因为复杂图会快速失败；训练图确认适配后再显式开启。

### 修正 2：把 result_key 合约前移

目标：继续坚持“key 级契约和流程健康检查尽量在运行前发现”。

修改方向：

```text
config_schema._validate_node_async_fields():
  async="result_key" 时检查 result_key 必须在 provides 中。

graph_config._parse_node():
  直接 parse 时也执行同样检查，避免绕过 schema。

runtime._run_async_node():
  保留最后防线，但不再作为主要报错位置。
```

验收标准：非法 result_key 在 `validate` / `run_checked` preflight 阶段失败，不进入 runtime。

### 修正 3：detached async 增加 flush timeout

目标：detached 用于 logging/metrics/diagnostics 时，后台任务卡死不应拖死训练收尾。

修改方向：

```text
RuntimeOptions 增加 async_flush_timeout: float | None。
detached flush 使用 future.result(timeout=async_flush_timeout)。
timeout 时记录 async_detached_timeout，不让主流程失败。
executor shutdown 对 timeout 场景使用 wait=False / cancel_futures=True。
```

注意：Python 线程不能被安全强杀；这里的目标是不无限等待 runtime 收尾。业务 writer 仍应自己支持超时、停止信号或队列关闭。

### 修正 4：支持 async nodeset，补齐子流程级异步

目标：让异步表达从“单 node 后台执行”扩展到“流程图中的一段子流程异步执行”。

修改方向：

```text
允许 nodeset call-site 使用 async="detached" 或 async="result_key"。

detached nodeset:
  后台运行子 runtime。
  输入按引用传入。
  完成后不回写父 context。
  失败/超时只记录 async_detached_failed / async_detached_timeout。

result_key nodeset:
  后台运行子 runtime。
  join 时只把 result_key 从子 context 写回父 context。
  result_key 必须在 call-site provides 中，并且必须可由 nodeset exports 导出。
```

仍然不做：

```text
不自动 merge async context。
不回写多个 key。
不做自动并行调度器。
不替业务对象处理线程安全。
```

示例目标：

```text
b -> async(nodeset.e_f) provides f_result
c requires f_result
```

### 修正 5：深化 block execution，但明确当前版本边界

目标：不要把当前保守 runner 当成最终 compiled block；逐步接近手写训练 loop。

短期文档修正：

```text
当前 execution="block" 是 conservative block runner。
新增 `execution="compiled"` 后，低开销线性段由 `CompiledBlock` 预分析并按 block 执行；不满足条件的图回退 plan。
```

实现方向：

```text
新增内部 LinearBlock / CompiledBlock 预分析。
已对无 async、无 nodeset、无多出边的线性段启用。
线性段内按预绑定 NodeFrame 顺序直接执行。
跳过 ready queue、edge activation 和重复 requirements 扫描。
trace="boundary" + node_hooks=False 时只记录 block 边界和失败。
遇到 decision/cycle/async/nodeset/node hook 开启时回退当前 frame runner。
```

仍然不做：任意 Python 源码生成、复杂 CFG 优化、自动并行、自动 context merge。

### 修正 6：彻底移除 CONTRACT.examples 输出内容审计

目标：保持 example 的最小可运行价值，同时不让 VibeFlow 退回数据内容审计。

修改方向：

```text
CONTRACT.examples 只用于证明 node 可用最小输入/参数运行。
继续检查：
  inputs / params 是 mapping。
  example inputs 覆盖 CONTRACT.requires。
  run_pure 可运行。
  run_pure 返回 Mapping。
  返回 key 等于 CONTRACT.provides。
不再检查：
  outputs 是否必填。
  actual_outputs == expected_outputs。
  actual_outputs 是否 JSON serializable。
  output_schema[key].snapshot == "opaque" 特例。
```

`outputs` 字段从 `CONTRACT.examples` 结构中移除；示例只声明 `inputs` 和 `params`。JSON-safe 约束只保留在 JSONC 配置、node 默认参数、报告和 trace 这些 artifact 格式上，不应用到 runtime value。

### 修正 7：预绑定 runtime hook callable

目标：减少高频 hook 调用的动态查找成本，并显式支持 run/node/nodeset/block/failure hook 粒度。

修改方向：

```text
PipelineRuntime.__init__ 或 ExecutionPlan 构建时生成 hook table：
  before_run
  after_run
  run_failed
  before_node
  after_node
  node_failed
  before_nodeset
  after_nodeset
  nodeset_failed
  before_block
  after_block
  block_failed

表里只保存实际 callable。
对应 hook 粒度开关为 False 时不构造或不调用该层 hook table。
运行时不再每次 getattr(plugin, hook)。
```

训练 preset 默认只保留 run+block hook；debug preset 保留全部 hook。

### 推荐执行顺序

```text
1. 先做 CLI/runtime options、result_key 前移、detached timeout。
   这三项小、收益直接，修复正式训练入口和卡死风险。

2. 再做 async nodeset。
   这是最大语义缺口，用于补齐 async(e -> f) 子流程表达。

3. 再彻底移除 CONTRACT.examples 输出内容/JSON snapshot 审计。
   只保留 example 可运行和输出 key 合约检查。

4. 最后做 hook callable 预绑定和 block 深化。
   它们是性能深化，不影响当前正确性闭环。
```
