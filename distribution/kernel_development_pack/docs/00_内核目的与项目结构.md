# 00. 内核目的与项目结构

Topology Kernel 的目的不是让开发者写得更自由，而是故意限制程序结构，强制程序长期保持低耦合、可审计、可维护。

它适合人机协同开发，尤其适合让 LLM 参与长期项目开发。LLM 很容易把功能越写越大，或者在出问题时用补丁覆盖问题。这个内核要求所有业务能力都拆成小型纯函数 node，并用 JSONC 配置组织拓扑。只要 node 或配置违反规则，运行前健康检查会拒绝执行并给出原因。

## 使用者需要开发什么

通常只需要开发这些目录：

```text
project/
  nodes/          # 业务 node，只允许纯函数逻辑
  base_lib/       # 可复用纯函数 helper
  plugins/        # 策略、编译、运行、边界 hook 插件
  configs/        # JSONC 拓扑配置和 nodeset 定义
  registry.py     # 注册 node 和 boundary
  boundaries.py   # 真实副作用边界，如文件、网络、数据库调用
```

内核目录建议放在：

```text
kernel/topology_kernel/
```

启动脚本建议放在项目根目录：

```text
run.py
```

## 标准工作流

1. 写一个 node，补全 `NODE_INFO`、`CONTRACT`、`run_pure(self, inputs, params)`。
2. 在 `project/registry.py` 注册 node，同时提供配置 schema 和默认值。
3. 在 `project/configs/*.jsonc` 中用不同 `name` 调用 node，并按需写 `config`。
4. 如果需要组合大模块，写 `nodesets`，再用 `type: "nodeset.xxx"` 调用。
5. 运行健康检查。
6. 只有健康检查通过后，才执行运行。
7. 导出 Mermaid 图审查整体结构。

## 常用命令

在复制了模板的项目根目录运行：

```powershell
python run.py validate --config project/configs/main.jsonc
python run.py run --config project/configs/main.jsonc --run-root runs
python run.py mermaid --config project/configs/main.jsonc --output reports/graph.mmd
python run.py inspect-config --config project/configs/main.jsonc
python run.py quality --path project
```

`validate` 和 `run` 都会执行强制健康检查。`run` 在检查失败时会拒绝运行，并把报告写入运行目录。
