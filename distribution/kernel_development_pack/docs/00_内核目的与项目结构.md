# 00. 内核目的与项目结构

VibeFlow 的目的不是让开发者写得更自由，而是故意限制程序结构，强制程序长期保持低耦合、可审计、可维护。

它适合人机协同开发，尤其适合让 LLM 参与长期项目开发。LLM 很容易把功能越写越大，或者在出问题时用补丁覆盖问题。这个内核要求所有业务能力都拆成小型、可审计的 node，并用 JSONC 配置组织成标准流程图。普通 node 保持纯函数；真实 IO 只能出现在内核按 `flow_kind` / `external` 派生并审计的 effect scope 中。只要 node 或配置违反规则，运行前健康检查会拒绝执行并给出原因。

## 使用者需要开发什么

通常只需要开发这些目录：

```text
project/
  nodes/          # 业务 node；普通 node 纯函数，特定 flow_kind 可有受控 IO
  base_lib/       # 可复用纯函数 helper
  plugins/        # policy/compiler/runtime 插件
  configs/        # JSONC 拓扑配置和 nodeset 定义
  registry.py     # 注册 node/base_lib/plugin 可用资源
```

内核建议作为单文件归档放在：

```text
kernel/
  vibeflow-kernel.zip
  MANIFEST.sha256
  README.md
  docs/
  tools/
    mermaid-renderer/
  THIRD_PARTY_NOTICES.md
```

业务开发时不需要解包它，也不应修改 `kernel/` 下随内核分发的文档、工具配置和许可说明。所有需要直接使用的公开对象都通过 `from vibeflow import ...` 暴露，例如：

- `NodeInfo`、`NodeContract`：编写 node 元数据和契约。
- `NodeRegistry`：注册项目 node。
- `HealthFinding`：插件追加健康检查时使用。
- `RuntimeOptions`、`run_checked`：在自定义启动器中运行内核时使用。

如果需要 SVG 图输出，分发包还会包含：

```text
kernel/tools/mermaid-renderer/
```

启动脚本建议放在项目根目录：

```text
run.py
```

## 标准工作流

开始前先判定任务类型，不要把新建项目和修改已有项目混成同一套流程。

### 新建项目（greenfield）

1. 把需求先表达为粗粒度的 planned flow，必要时用独立 planned nodeset 逐层细化。
2. 在真实 `project/configs/*.jsonc` 中声明 planned 调用点的 `id`、`display_name`、`description`、`flow_kind` 和数据契约；只有复用既有 implemented 类型时才在审核前写 `type_used`。
3. 用显式 `pipeline.edges` 组织待审核 flow，必要时将大块拆成独立 planned nodeset；此阶段不实现 Python node。
4. 用 `python run.py review --config ... --output ...` 更新登记架构文档、完成正式校验并生成 canonical expanded SVG，供人类审核。
5. 只有人类在后续消息中明确批准后，才为 planned 部分实现 `NODE_INFO`、`CONTRACT`、`run_pure(self, inputs, params)` 并在 `project/registry.py` 注册，最后再做 runtime probe。

### 修改已有项目（existing）

1. 先读取所属 root 登记的 `ARCHITECTURE.jsonc`，再沿其 source 定位真实 workflow config、nodeset 和 registry。
2. 修改前列出`复用 / 修改 / 删除 / 新增`清单，写明 source path、node/edge id、现有职责和计划变化。清单未列出的 id、edge、hook 和调用层级默认保持。
3. 在登记的同一个 workflow 及其导入的 nodeset 上原位修改；只把真正待审核的新增或变更部分标为 `planned`。
4. 不得为了审核新建平行 config，也不得用概念图或手写 Mermaid/SVG 替代真实 config 审核。
5. 使用同一条 `run.py review` 流程审核；如果用户要求“审核后再实现”，本次 review 成功也不是批准，必须等待后续明确消息。

Runtime 默认审计流程而不是数据内容：node 间可以按引用传递 tensor、model、optimizer、batch 等 Python 对象；报告和 trace 只记录 key、事件、summary 和失败位置，不保存真实对象内容。

## 分发包里不要改什么

这些文件由构建脚本生成，业务开发者和 AI 不应修改：

- `kernel/vibeflow-kernel.zip`
- `kernel/MANIFEST.sha256`
- `kernel/README.md`
- `kernel/docs/`
- `kernel/tools/mermaid-renderer/package.json`
- `kernel/tools/mermaid-renderer/package-lock.json`
- `kernel/THIRD_PARTY_NOTICES.md`
- `run.py`

`run.py` 会在启动时校验这些内核和启动器文件。如果有人把内核源码重新解包到 `kernel/vibeflow/`，校验会把它当作 unexpected 文件并拒绝继续运行。根目录的 `README.md`、`AGENTS.md` 以及项目自己的说明文档是给使用者定制项目流程用的，不属于内核完整性校验范围，可以按项目需要修改。

分发包不内置 `.gitignore`，Git 忽略策略由项目自行决定。常见建议包括忽略 `kernel/tools/mermaid-renderer/node_modules/`、`runs/`、`reports/`、`__pycache__/` 和 `*.pyc`。

## 常用命令

在复制了模板的项目根目录运行：

```powershell
python run.py validate --config project/configs/main.jsonc
python run.py review --config project/configs/main.jsonc --output reports/graph.expanded.svg
python run.py run --config project/configs/main.jsonc --run-root runs
python run.py delegate-cli --config project/configs/main.jsonc -- --input data.yaml --verbose
python run.py mermaid --config project/configs/main.jsonc --output reports/graph.mmd
python run.py ascii --config project/configs/main.jsonc --output reports/graph.txt
python run.py svg --config project/configs/main.jsonc --output reports/graph.svg
python run.py svg --config project/configs/main.jsonc --expand-nodesets --output reports/graph.expanded.svg
python run.py inspect-config --config project/configs/main.jsonc
python run.py quality --path project
```

首次使用 `svg` 命令前，需要在 `kernel/tools/mermaid-renderer/` 执行 `npm install`。

不要求系统预装 Google Chrome。正常执行 `npm install` 后，Puppeteer 会安装并使用自己的浏览器缓存；如果该缓存不可用，VibeFlow 会再尝试非 snap 的系统 Chrome/Chromium。`/snap/bin/chromium` 会被跳过，因为它在 Puppeteer/mermaid-cli 下常见 profile lock 启动失败。

正式架构审核使用 `python run.py review`，它会生成 canonical expanded SVG 并在发布前检查其结构。单独的 `svg` 命令仍可用于诊断或图形导出。`python run.py mermaid --expand-nodesets --output reports/graph.expanded.mmd` 只生成 Mermaid 源码调试文件；Mermaid CLI/mmdc 是 VibeFlow 内部渲染实现，不是公开审核入口。

`validate` 和 `run` 都会执行强制健康检查。`run` 在检查失败时会拒绝运行，并把报告写入运行目录。

需要把 workflow 当成普通业务命令行程序时，使用 CLI 让渡模式 / `delegate-cli`。它把业务参数作为 `cli.argv` 传入图，从唯一 `cli.exit_code` 输出取 `0..255` 退出码，并保留真实 stdin/stdout/stderr。首个 `--` 可用来明确分隔 core 参数与业务参数，也可省略。VibeFlow 自身诊断写入当次 run 目录的 `vibeflow.log`，不污染业务标准流。`delegate-cli` 不替代通用执行命令 `run`，也不改变正式架构审核命令 `review`。

## 关键变化

- 旧 `boundary` 模型已移除。
- 旧 `pipeline.loops` / pipeline 或 edge 级 `max_iterations` / edge `max_executions` 已移除；训练循环使用一等 loop node 的 `loop.max_iterations`。
- `flow_kind=io` 对应 `effect_scope=terminal`，可使用真实标准流、`print` / `input` / `argparse`。
- `flow_kind=document` / `data_store` 对应 `effect_scope=python_io`，可使用文件、环境、网络、数据库、subprocess 和终端。
- 普通 implemented node 和 planned `python_stub` 对应 `effect_scope=none`。
- 任意 `flow_kind` 只要 `NodeInfo.external=True`，即以最高优先级使用 `effect_scope=trusted`；plugin 也是 `trusted`。它们仍须遵守契约、拓扑、输出和 trace。
- 图形 `flow_kind=terminal` 仍是 `effect_scope=none`，不等于权限档位 `terminal`。effectful / external node 的 examples 只做结构检查，不执行。
- 训练/批处理类 side task 可显式使用 `async: "detached"` 或 `async: "result_key"`，但 runtime 不自动合并异步 context。
