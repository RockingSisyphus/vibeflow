# 02. 注册与配置默认值

所有 node 都必须在 registry 中注册。注册时必须提供：

- registry key，也就是 config 中使用的 `type`。
- node class。
- `config_schema`。
- `config_defaults`。

即使 node 没有配置项，也必须显式写空对象。

## 注册示例

```python
from vibeflow import NodeRegistry
from nodes.math_nodes import AddNode, EndNode, SeedNode, StartNode


def build_node_registry() -> NodeRegistry:
    registry = NodeRegistry()
    registry.register("demo.start", StartNode, config_schema={}, config_defaults={})
    registry.register(
        "demo.seed",
        SeedNode,
        config_schema={"value": {"type": "number"}},
        config_defaults={"value": 1},
    )
    registry.register(
        "demo.add",
        AddNode,
        config_schema={"delta": {"type": "number"}},
        config_defaults={"delta": 1},
    )
    registry.register("demo.end", EndNode, config_schema={}, config_defaults={})
    registry.register(
        "demo.no_config",
        SomeNode,
        config_schema={},
        config_defaults={},
    )
    return registry
```

## 支持的配置类型

配置 schema 采用轻量 JSON-like 结构。每个字段 schema 至少要有 `type`：

```jsonc
{
  "delta": {"type": "number"},
  "label": {"type": "string"},
  "enabled": {"type": "boolean"},
  "options": {"type": "object"},
  "items": {"type": "array"}
}
```

当前常用类型：

- `any`
- `number`
- `integer`
- `string`
- `boolean`
- `object`
- `array`
- `null`

`bool` 不会被当成 `integer` 或 `number` 接受。

## 默认值必须完整

`config_defaults` 必须和 `config_schema` 字段完全一致：

```python
config_schema={"delta": {"type": "number"}}
config_defaults={"delta": 1}
```

以下写法会失败：

```python
config_schema={"delta": {"type": "number"}}
config_defaults={}
```

以下写法也会失败：

```python
config_schema={}
config_defaults={"delta": 1}
```

这样做的目的是保证 node 在没有调用处配置时也能运行，并且所有配置项都可审计。

## 调用处覆盖配置

```jsonc
{
  "name": "add_small",
  "type": "demo.add",
  "requires": ["value.in"],
  "provides": ["value.small"],
  "config": {"delta": 1}
}
```

同一 node_type 可以多次调用：

```jsonc
{
  "name": "add_big",
  "type": "demo.add",
  "requires": ["value.in"],
  "provides": ["value.big"],
  "config": {"delta": 10}
}
```

注意：不同调用点的配置彼此独立，但输出 key 仍然必须符合该 node 的契约和图的 provider 规则。如果需要同一个 node_type 输出不同 key，必须设计对应 node 类型或后续引入明确的契约映射机制。
