# 08. 给 AI 开发者的约束清单

当你作为 AI 使用这个内核开发程序时，必须遵守下面的清单。

## 首要边界

- 不要解包或阅读 `kernel/vibeflow-kernel.zip`。
- 不要修改 `kernel/`、`run.py` 或 `kernel/MANIFEST.sha256`。
- 根目录 `README.md`、`AGENTS.md` 和项目说明文档可以按项目需要自定义；修改它们不应绕过或改变内核规则。
- 只在 `project/nodes/`、`project/base_lib/`、`project/plugins/`、`project/configs/` 和必要的项目文档中工作。
- 需要 API 时使用文档说明的公开入口：`NodeInfo`、`NodeContract`、`NodeRegistry`、`HealthFinding`、`RuntimeOptions`、`run_checked`。

## 开发 node 前

- 先确认这个功能能否拆成一个小纯函数 node。
- 先确定它在标准流程图中的 `flow_kind`。
- 如果需要文件、网络、数据库、浏览器或外部系统，不要直接写进 node；用 `io`、`data_store`、`document` 或外部系统输入/输出契约建模。
- 如果只是包装第三方库或外部维护代码，使用真实 `flow_kind` 并设置 `external=True`。
- 如果逻辑可复用，放到 `base_lib/` 的纯函数里。
- 如果多个 node 形成模块，写 nodeset，不要让 node 互相调用。

## 写 node 时

- 必须写完整 `NODE_INFO`。
- `NODE_INFO.flow_kind` 必须是标准类型，不能省略。
- 必须写完整 `CONTRACT`。
- 必须使用 `run_pure(self, inputs, params)`。
- 只读取 `inputs` 和 `params`。
- 只返回 `CONTRACT.provides` 声明的 key。
- 不修改 `inputs`。
- 共享对象原地更新只能作为显式业务语义使用，并通过输出 key 暴露更新后的引用。
- 不使用动态 output key。
- 不写 public helper 方法。
- 不在 `__init__` 中接收资源或保存外部对象。
- 不导入其他 node。

## 写 decision 时

- 必须声明 `flow_kind="decision"`。
- 必须输出 route-like key，例如 `flow.route`。
- 建议在 `output_schema` 中声明 enum / boolean 分支。
- 从 decision 出发的 edge 必须写 `when`。
- 每个非回环分支必须能走到 terminal end。

## 注册 node 时

- 必须提供 `config_schema`。
- 必须提供 `config_defaults`。
- 没有配置也写 `{}`。
- 默认值必须覆盖所有 schema 字段。
- 默认值必须是 JSON snapshot safe，不要把 tensor/model/函数/文件句柄放进 config。

## 写 config 时

- 每个调用点都要有唯一 `id`，并用 `type_used` 指向 Python node `NodeInfo.type_key`、nodeset `type_key` 或系统类型。
- 同一个实现类型多次调用时，用不同 `id` 和不同 `config`。
- 每个 `pipeline.nodes[]` 调用点都必须写 `display_name` 和 `description`；不要依赖 Python `NODE_INFO` 替代调用点说明。
- `display_name`、`description`、`style`、`similar_to` 是调用点元数据，不进入运行时 `params`。
- 如果运行时确实需要同名参数，只能写进 `"config": {...}`，不要把可视化元数据塞进 node 参数。
- 颜色只写在 `style` 下，且只能使用 `fill`、`stroke`、`text` 三个 `#RRGGBB` 字段。
- 不得使用系统保留色作为自定义颜色：普通 node `#ECECFF/#9370DB/#333333`，planned `#fef08a/#ca8a04/#713f12`，plugin resource `#eff6ff/#2563eb/#1e3a8a`，base_lib resource `#ecfdf5/#059669/#064e3b`，health、external、document、nodeset 等语义色也都禁止复用。
- 只有当两个 node 确实是有意变体或副本时，才写 `similar_to`；必须指向同作用域已存在的 node，`relationship` 只能是 `variant` 或 `copy`，并写清 `reason`。不要用它掩盖应拆分或复用 base_lib 的重复实现。
- `requires` / `provides` 要和 node 契约一致，并且每个 contract 对象都必须写非空 `display_name`。
- Mermaid/SVG 节点内只显示易读名、`id`、`type_used`、状态和说明；contract 会显示在已有连边上，先显示 contract `display_name`，不要为了图里有 contract 而把它写进 `description`。
- `config` 只能覆盖 registry schema 中声明的字段。
- 顶层通用设置写 `global_config`；它会传入所有 node 的 `params["_global"]`，同名 schema 字段会覆盖实际 params。
- 当 `global_config` 或 nodeset 调用 `config` 覆盖局部配置且 `allow_config_override` 为 `false` 时，健康检查应出现 warning；不要为了消除 warning 隐藏覆盖关系。
- node 导入的 `base_lib` 必须在所属 root 的 `vibeflow_project.jsonc.base_lib.modules` 中声明为 `implemented`；planned base_lib 只做规划占位。
- 只有所属 root 的 `vibeflow_project.jsonc.plugins` 中显式声明的 plugin 会启用；planned plugin 不加载、不执行 hook。
- 每个 `base_lib.modules[]` 和 `plugins[]` 对象声明都必须写 `display_name` 和 `description`；不要只写字符串、`module` 或 `path`。`PLUGIN_INFO` / `BASE_LIB_INFO` 是源码级元数据，不能替代 config 中本次启用资源的用途说明。
- planned node/nodeset 默认是 `blocking`；需要设计期连通性才写 `planned_behavior: "transparent"`。只有开发测试 stub 才写 `python_stub`，且 stub 必须在 `project/stubs/` 下。
- 控制流必须写显式 `pipeline.edges`。
- 不要依赖 `requires/provides` 自动连边；它们只校验和解释已声明 edge。没有显式 edge，就没有图上的边。
- Health 会推断同步主线、data bypass 和 async 相关边。主线 edge 负责调度并在 SVG/Mermaid 加粗；data bypass edge 只投递数据不触发目标并显示虚线；async edge 连接显式 async node/nodeset，不进入同步主线。
- 非 decision 同步 fan-out 必须有明确语义：分支通过 `join_policy: "all"` 汇合、被识别为 data bypass，或分支目标显式写 `async`。否则按 `GRAPH.MAINLINE.*` 的 details 修 `source`、`target`、`branch_nodes` 和 `branch_edges`。
- 每个可执行 pipeline 和 nodeset 内部 pipeline 都要有 terminal start/end。
- 普通 `pipeline.edges` 和 nodeset 内部 `pipeline.edges` 不允许形成环；出现环会触发 `GRAPH.CYCLE.FORBIDDEN`。
- `decision` 只用于分支选择，不能用 decision cycle 表达 retry / again / done；训练循环、多 batch、多 epoch、carry state 和 metrics collect 必须使用唯一一等 loop 类型 `vibeflow.loop.while`。
- loop node 的 `loop` 字段是执行语义元数据，不进入运行时 `params`；body 必须指向 nodeset，必须写 `max_iterations`，且 `stop_after` / `stop_when` 必须二选一。
- 不要再写 `vibeflow.loop.for_each`、`loop.items`、`loop.epochs` 或 `loop.until`；batch/epoch/遍历状态应放进 body nodeset，用 counter/index/batch selector 节点和 `loop.carry` 表达。
- `stop_after` 必须是正整数且不能大于 `max_iterations`；`stop_when.from` 必须解析到 body output 或 loop state 中的 bool。
- `execution="block"` 是严格 block 模式；如果 loop/body 不能 block 化会启动前报错。`execution="compiled"` 是性能模式，会优先生成 graph/nodeset/loop block，不能生成的区域回退 plan。
- loop 在 Mermaid/SVG 中使用 trapezoid (`trap-b`) 形状和默认 `loopNode` 系统样式；`loopNode` 默认颜色是系统保留色，不能作为自定义 `style` 颜色；如需改 loop 颜色，只能写其他非保留 hex 色。
- join 默认是 safe OR。互斥分支汇入一个 node 可用默认 `safe_any`；明确任一 active edge 都可触发时才写 `join_policy: "any_active"`；需要等待所有 incoming 时写 `join_policy: "all"`。
- 不要让 conditional provider 和 unconditional provider 同时可能满足同一个 `exactly_one` requirement；这会触发 `GRAPH.JOIN.AMBIGUOUS_UNCONDITIONAL`。复杂汇合优先写显式 merge/select node。
- 使用 `max_steps` 作为顶层运行时安全护栏；loop 的防护写 `loop.max_iterations`。
- nodeset 内部配置覆盖使用 `node_configs`。
- nodeset 的调用 `config` 覆盖 nodeset 内部 `global_config`，再覆盖内部 node 局部 config；精确覆盖内部 node 时再用 `node_configs`。
- 嵌套 nodeset 和 `vibeflow.loop.while` 使用 dotted path，例如 `outer.inner_node`、`train_loop.step_node`；loop 路径段写调用点 `id`，会进入该调用点的 `loop.body`，不要直接写 body `type_key`。
- nodeset 必须是独立 JSONC 文件，根对象声明 `type_key`、`display_name`、`description`、`requires`、`provides` 和 `pipeline`；主 config 不允许内联 `nodesets`。
- nodeset 文件可以写 `nodeset_imports` 调用更深层 nodeset；调用处直接把 nodeset `type_key` 写进 `type_used`，不要写旧 `nodeset.xxx` 前缀。
- nodeset 调用和 `loop.body` 都构成 nodeset dependency，不能直接或间接递归；出现 `NODESET.RECURSION` 时拆结构或使用真正的 `vibeflow.loop.while` 循环语义。
- 为消除 `NODESET.SMELL.TOO_WIDE` 拆成更多小 nodeset 是推荐方向；如果怀疑解析慢，用 `VIBEFLOW_CONFIG_TRACE=1 python run.py validate --config ...` 查看 import 和 nodeset 解析耗时。
- 异步 side task 只能显式写 `async: "detached"` 或 `async: "result_key"`；复杂后台任务可把调用点写成 nodeset `type_used` 并设置 `async`。不要在 node 里私自启动线程或自动合并 context。
- 不要使用旧 `pipeline.loops`、pipeline/edge 级 `max_iterations`、edge `max_executions` 或 `boundary`。

## 写 plugin 时

- plugin 配置路径相对当前 config 文件。
- implemented plugin 必须提供 `PLUGIN_INFO`；planned plugin 至少写 `name`、`type`、`status`、`display_name` 和 `description`。
- 插件设置写在 `config` 或 `settings`，插件通过 `plugin.config` 或 `configure(config)` 接收。
- 默认类名是 `Plugin`；如果不是，必须在 config 中写 `class`。
- policy plugin 放宽规则时必须声明 `relaxations`。
- health hook 返回值必须是 `HealthFinding` 列表或元组。
- runtime hook 只做观测、记录和进度上报，不要用它替代显式 `io` / `data_store` / `document` 节点。

## 出错时

优先根据健康报告修正根因：

- schema 错误：修 config 结构。
- topology 错误：修显式 edge、start/end、cycle、nodeset。
- flow_kind 错误：修 `NodeInfo.flow_kind` 或 planned config `flow_kind`。
- decision 错误：修 route output、output schema 或 edge `when`。
- purity 错误：拆 node、抽纯 helper、删除隐藏副作用。
- config 错误：修注册默认值或调用处 `config`。
- base_lib 错误：拆纯 helper，删除副作用和过深依赖。
- external 相关错误：确认是否应该 `external=True`，但不要用它绕过契约和拓扑规则。
- 所有 warning/error 先读 `object_type`、`object_id`、`source_location` 和 `details`；`details.owner` 区分顶层 `pipeline` 与 `nodeset:<name>`，flow/data 规则会给 incoming/outgoing edge、direct source、provider type 或 downstream requirement 摘要，mainline 规则会给 `source` / `target`、`branch_nodes`、`branch_edges`、`mainline_path` 和 `suggested_fixes`。
- 如果 `details.aggregated == true`，先修代表 finding 指向的 `owner` / `node` / `required_type` / `direct_sources`；`occurrences` 是 nested nodeset / loop body 静态展开后被压缩的重复次数，不是独立 bug 数。
- `quality` / `quality-check` 的文本报告也会打印 `object_id` 和紧凑 `details:`；重复函数、依赖环、双向依赖、跨目录/内部模块 import 要优先按 details 中的函数和 import site 修改。
- duplicate logic warning：先看报告 `details.nodes`、`details.node_types`、`details.fingerprint` 和 `details.duplicate_group`；如果是有意变体/副本，再用 `similar_to` 明确声明。
- nested nodeset/loop trace 问题：顶层顺序看 `runtime.exec_order`，完整嵌套顺序看 `runtime.qualified_exec_order` 和 event 的 `path` 数组；不要解析 dotted 字符串作为机器接口。

不要通过扩大 node、加补丁、伪造 `flow_kind` 或放宽 policy 来绕过硬规则。
