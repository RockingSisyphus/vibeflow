# 08. 给 AI 开发者的约束清单

当你作为 AI 使用这个内核开发程序时，必须遵守下面的清单。

## 首要边界

- 不要解包或阅读 `kernel/vibeflow-kernel.zip`。
- 不要修改 `kernel/`、`run.py` 或 `kernel/MANIFEST.sha256`。
- 根目录 `README.md`、`AGENTS.md` 和项目说明文档可以按项目需要自定义；修改它们不应绕过或改变内核规则。
- 只在 `project/nodes/`、`project/base_lib/`、`project/plugins/`、`project/configs/` 和必要的项目文档中工作。
- 需要 API 时使用文档说明的公开入口：`NodeInfo`、`NodeContract`、`NodeRegistry`、`HealthFinding`、`RuntimeOptions`、`run_checked`。
- 在理解、解释或修改项目架构前，必须优先阅读所属 root 登记的 `ARCHITECTURE.jsonc`，用它掌握入口流程、nodeset 调用层级、节点职责、数据契约和配置来源。它是生成且不可执行的审查文档；要改变架构，只能修改真实 workflow config 或其导入的相关 nodeset JSONC，必要时再修改 registry metadata/config schema，不能手工编辑生成文件。单独的 `architecture` 命令只用于缺失文档的预读修复或单项诊断；正式审核必须使用会自动更新文档的 `python run.py review --config ... --output ...`。
- root 运行配置只写在 `vibeflow_project.jsonc`：`runtime.async_max_workers` 是每个 Runtime 自有线程池的并发数（默认 4），`runtime.async_flush_timeout` 是 detached task 收尾等待时间（默认 `null`，可设非负秒数），`runtime.nodeset_max_depth` 是普通 nodeset 与 `loop.body` 的最大静态嵌套深度（默认 4）。不要在 workflow config 中伪造这些 root 选项，也不要寻找不存在的 worker/depth CLI 参数。

## 任务判定与审核协议

- 先明确判定任务是新建项目（greenfield）还是修改已有项目（existing）。只有 greenfield 或用户明确批准整体重构时，才从 `a → b → c` 粗粒度 planned 拓扑开始。
- existing 任务必须先读所属 root 登记的 `ARCHITECTURE.jsonc`，再按其 source 定位同一个真实 workflow config、相关 nodeset 和 registry。
- 修改前先输出`复用 / 修改 / 删除 / 新增`清单，对每项写明 source path、node/edge/hook id、当前职责和计划变化。只修改清单内对象；未列出的 id、edge、hook 和调用层级默认保持。
- existing 任务只能在登记的 workflow 及其真实 source 上原位修改。不得为审核新建平行 config，不得用手写 Mermaid/SVG、概念图或笼统差异图代替修改原项目。
- 正式审核统一执行 `python run.py review --config <登记的 workflow> --output <expanded.svg>`。它会做源图 preflight，自动更新登记的架构文档，再执行正式 validate、canonical expanded SVG 渲染和结构检查。
- `review` 是 fail-closed 的。任何阶段失败必须停止；不得直接调用 Mermaid CLI/mmdc，不得用 expanded `.mmd`、手写 SVG 或旧产物补位，也不得宣称审核成功。
- 用户说“审核后再实现”、“等我确认”或同类要求时，当前轮只能完成审核产物并停止。只有后续一条明确的用户批准消息才算通过；`validate`、`architecture`、`svg` 或 `review` 成功都不等于人类批准。

## CLI 让渡模式 / delegate-cli

- 用户要求“像普通命令行程序一样运行”时，使用 CLI 让渡模式 `python run.py delegate-cli --config project/configs/main.jsonc -- <business argv>`；不另写顶层 launcher 绕过 manifest、workspace、health 和 run artifact。
- 首个 `--` 是可选分界：分界前已知 core 参数由 VibeFlow 消费，未知 token 原序让渡；分界后全部原样让渡。业务参数与 core 同名时必须放到 `--` 之后。
- workflow 必须声明 `cli.argv` pipeline input；必须声明 `cli.exit_code` 的 `exactly_one` output requirement，并且最终唯一 provider 的 key/type 都是 `cli.exit_code`。其值必须是非 bool 整数 `0..255`。
- 业务 stdin/stdout/stderr 是真实进程标准流；不把它们包装成 `cli.response`，不依赖 VibeFlow 捕获/重放，不添加额外 JSON 或换行。
- VibeFlow 自身诊断必须去当次 run 的 `vibeflow.log`，不污染业务标准流；日志不得记录 argv 原文或业务流。只有 run 目录创建失败时可以向 stderr 写最小诊断并返回 1。
- 正常 `cli.exit_code` 和授权 `SystemExit` 原样返回 `0..255`；`SystemExit(None)` 为 0。只有 `io` / `document` / `data_store` node 或 runtime plugin 可发出授权 `SystemExit`。未授权或非法 code、health/runtime/CLI contract 失败返回 1；argparse 层已知 core 参数错误返回 2 且不创建 run。
- `delegate-cli` 不改变 `run` 的通用执行/结构化诊断职责，也不改变 `review` 的正式架构审核职责。

## 开发 node 前

- 先确认这个功能能否拆成一个小纯函数 node。
- 先确定它在标准流程图中的 `flow_kind`。
- 按固定映射选择副作边界：普通 implemented node 与 planned `python_stub` => `none`；`flow_kind=io` => `terminal`（真实标准流、`print` / `input` / `argparse`）；`document` / `data_store` => `python_io`（文件、环境、网络、数据库、subprocess、终端）；任意 `external=True` node 和 plugin => 最高优先级 `trusted`。
- 图形 `flow_kind=terminal` 仍是 `effect_scope=none`，不要和权限档位 `terminal` 混淆。`effect_scope` 是内核派生结果，不是 config 可自由选择的字段。
- 如果只是包装第三方库或外部维护代码，使用真实 `flow_kind` 并设置 `external=True`。它会得到 `trusted` 并跳过普通实现的 IO/purity 限制，确实是显式信任绕过；不要用它给本项目内部代码逃避检查。契约、`flow_kind`、拓扑、输出和 trace 仍检查。
- effectful 或 `external=True` node 的 `CONTRACT.examples` 只做结构检查，不执行；只有 `none` 范围的普通实现样例可执行。
- 如果逻辑可复用，放到 `base_lib/` 的纯函数里。
- 如果多个 node 形成模块，写 nodeset，不要让 node 互相调用。

## 写 node 时

- 必须写完整 `NODE_INFO`。
- `NODE_INFO.flow_kind` 必须是标准类型，不能省略。
- 必须写完整 `CONTRACT`。
- 必须使用 `run_pure(self, inputs, params)`。
- 普通 `none` node 只读取 `inputs` 和 `params`；effectful node 仅使用其 scope 明确开放的外部能力。
- 只返回 `CONTRACT.provides` 声明的 key。
- 不修改 `inputs`。
- 共享对象原地更新只能作为显式业务语义使用，并通过输出 key 暴露更新后的引用。
- 不使用动态 output key。
- 不写 public helper 方法。
- 不在 `__init__` 中接收资源或保存外部对象。
- 不导入其他 node。

## 写 decision 时

- 必须声明 `flow_kind="decision"`。
- 必须输出 route-like key，例如 `flow.route`。
- 建议在 `output_schema` 中声明 enum / boolean 分支。
- 从 decision 出发的 edge 必须写 `when`。
- 每个非回环分支必须能走到 terminal end。

## 注册 node 时

- 必须提供 `config_schema`。
- 必须提供 `config_defaults`。
- 没有配置也写 `{}`。
- 默认值必须覆盖所有 schema 字段。
- 默认值必须是 JSON snapshot safe，不要把 tensor/model/函数/文件句柄放进 config。

## 写 config 时

- 每个调用点都要有唯一 `id`，并用 `type_used` 指向 Python node `NodeInfo.type_key`、nodeset `type_key` 或系统类型。
- 同一个实现类型多次调用时，用不同 `id` 和不同 `config`。
- 每个 `pipeline.nodes[]` 调用点都必须写 `display_name` 和 `description`；不要依赖 Python `NODE_INFO` 替代调用点说明。
- `display_name`、`description`、`style`、`similar_to` 是调用点元数据，不进入运行时 `params`。
- 如果运行时确实需要同名参数，只能写进 `"config": {...}`，不要把可视化元数据塞进 node 参数。
- 颜色只写在 `style` 下，且只能使用 `fill`、`stroke`、`text` 三个 `#RRGGBB` 字段。
- 不得使用系统保留色作为自定义颜色：普通 node `#ECECFF/#9370DB/#333333`，planned `#fef08a/#ca8a04/#713f12`，plugin resource `#eff6ff/#2563eb/#1e3a8a`，base_lib resource `#ecfdf5/#059669/#064e3b`，health、external、document、nodeset 等语义色也都禁止复用。
- 只有当两个 node 确实是有意变体或副本时，才写 `similar_to`；必须指向同作用域已存在的 node，`relationship` 只能是 `variant` 或 `copy`，并写清 `reason`。不要用它掩盖应拆分或复用 base_lib 的重复实现。
- `requires` / `provides` 要和 node 契约一致，并且每个 contract 对象都必须写非空 `display_name`。
- Mermaid/SVG 节点内只显示易读名、`id`、`type_used`、状态和说明；contract 会显示在已有连边上，先显示 contract `display_name`，不要为了图里有 contract 而把它写进 `description`。
- `config` 只能覆盖 registry schema 中声明的字段。
- 顶层通用设置写 `global_config`；它会传入所有 node 的 `params["_global"]`，同名 schema 字段会覆盖实际 params。
- 当 `global_config` 或 nodeset 调用 `config` 覆盖局部配置且 `allow_config_override` 为 `false` 时，健康检查应出现 warning；不要为了消除 warning 隐藏覆盖关系。
- node 导入的 `base_lib` 必须在所属 root 的 `registry.py.build_base_lib_registry()` 中注册为可用，并在当前 workflow config 的 `base_lib.modules[]` 中按 id 显式引用；只注册但未引用不能满足 import allowlist。
- 只有当前 workflow config 的 `plugins[]` 中按 id 显式引用的 plugin 会启用；root registry 中只注册但未引用的 plugin 不加载、不执行 hook。
- base_lib/plugin 的 `display_name` 和 `description` 写在 `project/registry.py` 的 resource registry 注册项里；workflow config 只写实际使用的 id 和必要 config/settings。不要把未使用资源写进 config 或用 root 配置全局启用。
- planned node/nodeset 默认是 `blocking`；需要设计期连通性才写 `planned_behavior: "transparent"`。只有开发测试 stub 才写 `python_stub`，且 stub 必须在 `project/stubs/` 下。
- 控制流必须写显式 `pipeline.edges`。
- 不要依赖 `requires/provides` 自动连边；它们只校验和解释已声明 edge。没有显式 edge，就没有图上的边。
- Health 会推断同步主线、data bypass 和 async 相关边。主线 edge 负责调度并在 SVG/Mermaid 加粗；data bypass edge 只投递数据不触发目标并显示虚线；async edge 连接显式 async node/nodeset，不进入同步主线。
- 空 terminal start/end 不读取、不提供、不转发业务数据。data bypass 必须从真实拥有 target 所需 envelope type 的 provider 发出；它只投递数据，不触发 target。
- 非 decision 同步 fan-out 必须有明确语义：分支通过 `join_policy: "all"` 汇合、被识别为 data bypass，或分支目标显式写 `async`。否则按 `GRAPH.MAINLINE.*` 的 details 修 `source`、`target`、`branch_nodes` 和 `branch_edges`。
- 每个可执行 pipeline 和 nodeset 内部 pipeline 都要有 terminal start/end。
- 普通 `pipeline.edges` 和 nodeset 内部 `pipeline.edges` 不允许形成环；出现环会触发 `GRAPH.CYCLE.FORBIDDEN`。
- `decision` 只用于分支选择，不能用 decision cycle 表达 retry / again / done；训练循环、多 batch、多 epoch、carry state 和 metrics collect 必须使用唯一一等 loop 类型 `vibeflow.loop.while`。
- loop node 的 `loop` 字段是执行语义元数据，不进入运行时 `params`；body 必须指向 nodeset，必须写 `max_iterations`，且 `stop_after` / `stop_when` 必须二选一。
- 不要再写 `vibeflow.loop.for_each`、`loop.items`、`loop.epochs` 或 `loop.until`；batch/epoch/遍历状态应放进 body nodeset，用 counter/index/batch selector 节点和 `loop.carry` 表达。
- `stop_after` 必须是正整数且不能大于 `max_iterations`；`stop_when.from` 必须解析到 body output 或 loop state 中的 bool。
- `execution="block"` 是严格 block 模式；如果 loop/body 不能 block 化会启动前报错。`execution="compiled"` 是性能模式，会优先生成 graph/nodeset/loop block，不能生成的区域回退 plan。
- loop 在 Mermaid/SVG 中使用 trapezoid (`trap-b`) 形状和默认 `loopNode` 系统样式；`loopNode` 默认颜色是系统保留色，不能作为自定义 `style` 颜色；如需改 loop 颜色，只能写其他非保留 hex 色。
- `requires` 是数据需求，不是控制分支数。join 默认是 safe OR；明确任一 active edge 都足够时才写 `any_active`，只有至少两条本轮都会激活的真实并行 schedule incoming 才写 `all`。transfer-only bypass 不计入 `all`。
- `join_policy: "all"` 只用于本轮都会激活的真实并行 schedule 分支；顺序链、互斥分支和 transfer-only edge 都不满足条件。
- 遇到 `GRAPH.DATA.RUNTIME_REQUIREMENT_UNREACHABLE`、`GRAPH.DATA.NO_PAYLOAD_BYPASS`、`GRAPH.JOIN.ALL_DEPENDS_ON_TRANSFER_ONLY`、`GRAPH.JOIN.ALL_BRANCHES_MUTUALLY_EXCLUSIVE` 或 `GRAPH.JOIN.REDUNDANT_ALL` 时，不要通过多画 start shortcut 或增加 `all` 消除现象；同一 decision 的互斥分支不能用 `all` 汇合，应按 finding 中的 schedule/transfer incoming、provider、分支条件和修复建议改真实数据来源。
- 不要让 conditional provider 和 unconditional provider 同时可能满足同一个 `exactly_one` requirement；这会触发 `GRAPH.JOIN.AMBIGUOUS_UNCONDITIONAL`。复杂汇合优先写显式 merge/select node。
- 使用 `max_steps` 作为顶层运行时安全护栏；loop 的防护写 `loop.max_iterations`。
- nodeset 内部配置覆盖使用 `node_configs`。
- nodeset 的调用 `config` 覆盖 nodeset 内部 `global_config`，再覆盖内部 node 局部 config；精确覆盖内部 node 时再用 `node_configs`。
- 嵌套 nodeset 和 `vibeflow.loop.while` 使用 dotted path，例如 `outer.inner_node`、`train_loop.step_node`；loop 路径段写调用点 `id`，会进入该调用点的 `loop.body`，不要直接写 body `type_key`。
- nodeset 必须是独立 JSONC 文件，根对象声明 `type_key`、`display_name`、`description`、`requires` 和 `provides`；implemented nodeset 必须有完整 `pipeline`，planned nodeset 可以无 body 占位，也可以带 planned body 逐步细化。主 config 不允许内联 `nodesets`。
- planned body 会进入架构 JSONC、展开 Mermaid/SVG 及适用的 dependency、recursion、depth、planned-descendant 检查，但不会按 implemented body 执行；`python_stub` nodeset 运行时仍是单个 stub。
- nodeset 文件可以写 `nodeset_imports` 调用更深层 nodeset；调用处直接把 nodeset `type_key` 写进 `type_used`，不要写旧 `nodeset.xxx` 前缀。
- nodeset 调用和 `loop.body` 都构成 nodeset dependency，不能直接或间接递归；出现 `NODESET.RECURSION` 时拆结构或使用真正的 `vibeflow.loop.while` 循环语义。
- 普通 nodeset 和 `loop.body` 共用嵌套深度：顶层 pipeline 为 0，首次进入 body 为 1，默认最多 4 层，循环迭代不累计。所有已加载及 planned 定义都检查；需要更深结构时在所属 root 的 `runtime.nodeset_max_depth` 中显式提高，不要写进 workflow config。
- 为消除 `NODESET.SMELL.TOO_WIDE` 拆成更多小 nodeset 是推荐方向；如果怀疑解析慢，用 `VIBEFLOW_CONFIG_TRACE=1 python run.py validate --config ...` 查看 import 和 nodeset 解析耗时。
- 异步 side task 只能显式写 `async: "detached"` 或 `async: "result_key"`；复杂后台任务可把调用点写成 nodeset `type_used` 并设置 `async`。不要在 node 里私自启动线程或自动合并 context。
- 不要使用旧 `pipeline.loops`、pipeline/edge 级 `max_iterations`、edge `max_executions` 或 `boundary`。

## 语义边界、共享职责与门禁

- `io` 边界可以从真实标准流读写、用 `argparse` 解析 `cli.argv`，并做表示/词法适配；业务判断、语义合并、领域校验和错误身份仍必须归属于明确的 process/decision node。
- `io` 边界在进入共享业务判断前必须无损保留未知字段/值、缺失状态和异常身份；不得提前压成默认错误。
- 两个入口若声明共享语义，运行路径就必须汇入同一套真实执行的共享节点或 nodeset；共享 helper、相似命名或复制实现不能证明共享语义。
- 入口 guard 只能增加该入口特有的词法检查，并须保留规范化后的错误身份。错误输出节点只能封装既有错误，不能把所有错误覆盖成默认类别。
- 不要建立 raw input 到 output 的 data bypass 来绕过共享解析、转换、合并或校验职责。辅助 data bypass 不能替代业务主线，输出 I/O 也不能重新扫描原始输入计算业务结果。
- 外部交互要用显式 `io` / `data_store` / `document` node 及 contract 建模。`io` 可做真实终端交互；`data_store` / `document` 可做真实 Python IO。语义结果仍由负责该语义的 node 提供；不要仅为获得更宽权限而伪造 `flow_kind`。
- 推荐脊柱是 `terminal → input I/O → process/nodeset → output I/O → terminal`。内部语义结果与外部输出使用不同的明确 key/type，由 output I/O 无损适配。
- tagged value 的 tag 使用业务规范中的精确字面量，value 转换为匹配的 Python 原生类型；不缩写 tag，不把整数留作字符串。
- `validate` 和 `quality` 只是静态门禁。每个 runnable entry 还要做最小 runtime probe（运行时探针），检查业务结果 key、`value` 原生类型、`runtime.stop_reason == "completed"` 和 `runtime.qualified_exec_order`。
- 已登记 workflow 每次修改 source 后必须重新生成架构文档。`validate` / `run` 报 `ARCHITECTURE.DOCUMENT.*` 时，按消息中的 project config、workflow、document、差异 path 和修复命令处理；不要直接改生成文件。
- 需要交付架构审核结果时使用 `python run.py review --config ... --output ...`，不要用分开的命令成功或文件存在代替 review 的顶层 JSON 状态和 `published` 结果。
- required gate 的顶层 `OVERALL`/退出状态才是完成结论。局部 case 或某个维度 PASS 不代表总体 PASS；代码修改后旧结果不再新鲜，必须重跑。
- 最新 required gate 不是针对当前代码的顶层 PASS 时，只能报告字面量 `INCOMPLETE`。若失败后代码未变，不要重复运行同一门禁；先按失败项和完整报告修复。实验要求结构化完成声明时必须按协议填写，不能用自然语言绕过。

## 写 plugin 时

- plugin 配置路径相对当前 config 文件。
- implemented plugin 必须提供 `PLUGIN_INFO`；planned 资源不要写进 resource registry，需要计划占位时使用 planned node/nodeset。
- 插件设置写在 `config` 或 `settings`，插件通过 `plugin.config` 或 `configure(config)` 接收。
- 默认类名是 `Plugin`；如果不是，必须在 config 中写 `class`。
- policy plugin 放宽规则时必须声明 `relaxations`。
- health hook 返回值必须是 `HealthFinding` 列表或元组。
- plugin 是 `effect_scope=trusted`，可执行 Python IO；runtime plugin 还可在 `delegate-cli` 中发出授权 `SystemExit`。不要用 hook 隐藏本应由显式 `io` / `data_store` / `document` node 表达的业务主流。

## 出错时

优先根据健康报告修正根因：

- schema 错误：修 config 结构。
- topology 错误：修显式 edge、start/end、cycle、nodeset。
- flow_kind 错误：修 `NodeInfo.flow_kind` 或 planned config `flow_kind`。
- decision 错误：修 route output、output schema 或 edge `when`。
- purity 错误：拆 node、抽纯 helper、删除隐藏副作用。
- config 错误：修注册默认值或调用处 `config`。
- base_lib 错误：拆纯 helper，删除副作用和过深依赖。
- external 相关错误：确认是否真的属于外部维护实现；`external=True` 会显式绕过普通 IO/purity 限制，但不得绕过契约和拓扑规则。
- 所有 warning/error 先读 `object_type`、`object_id`、`source_location` 和 `details`；`details.owner` 区分顶层 `pipeline` 与 `nodeset:<name>`，flow/data 规则会给 incoming/outgoing edge、direct source、provider type 或 downstream requirement 摘要，mainline 规则会给 `source` / `target`、`branch_nodes`、`branch_edges`、`mainline_path` 和 `suggested_fixes`。
- 如果 `details.aggregated == true`，先修代表 finding 指向的 `owner` / `node` / `required_type` / `direct_sources`；`occurrences` 是 nested nodeset / loop body 静态展开后被压缩的重复次数，不是独立 bug 数。
- `quality` / `quality-check` 的文本报告也会打印 `object_id` 和紧凑 `details:`；重复函数、依赖环、双向依赖、跨目录/内部模块 import 要优先按 details 中的函数和 import site 修改。
- duplicate logic warning：先看报告 `details.nodes`、`details.node_types`、`details.fingerprint` 和 `details.duplicate_group`；如果是有意变体/副本，再用 `similar_to` 明确声明。
- nested nodeset/loop trace 问题：顶层顺序看 `runtime.exec_order`，完整嵌套顺序看 `runtime.qualified_exec_order` 和 event 的 `path` 数组；不要解析 dotted 字符串作为机器接口。

不要通过扩大 node、加补丁、伪造 `flow_kind` 或放宽 policy 来绕过硬规则。
