# 07. 启动命令与报告

推荐使用模板 `run.py`，因为它会先把 `kernel/vibeflow-kernel.zip` 加入 `sys.path`，然后把根目录 `vibeflow_config.jsonc` 传给内核 CLI。内核按 workspace roots 顺序读取每个 root 的 `vibeflow_project.jsonc`，再从同一个 `registry.py` 加载可用 node/base_lib/plugin registry 和 quality 范围。具体 workflow config 通过 id 引用实际启用的 base_lib/plugin。

不要直接修改 `run.py` 来绕过校验。它在导入内核前会先校验 `kernel/MANIFEST.sha256`，确认内核归档、启动器、`kernel/docs/`、renderer 配置和许可说明没有被改动。根目录 `README.md`、`AGENTS.md` 和项目自己的说明文档是使用者可定制内容，不属于内核完整性校验范围。

## workspace 配置

发布包根目录的 `vibeflow_config.jsonc` 是 workspace 索引。单项目模板默认只声明 `project/`：

```jsonc
{
  "policy": {},
  "roots": [
    {"id": "project", "path": "project"}
  ]
}
```

如果同一个仓库里同时有框架层和任务层，可以增加多个 root：

```jsonc
{
  "policy": {},
  "roots": [
    {"id": "vibetrain", "path": "vibetrain"},
    {"id": "project", "path": "project"}
  ]
}
```

每个 root 下放自己的 `vibeflow_project.jsonc`，声明 registry、quality 和可选 runtime 参数：

```jsonc
{
  "registry": "registry.py:build_node_registry",
  "quality_enabled": true,
  "runtime": {
    "async_max_workers": 4,
    "async_flush_timeout": null,
    "nodeset_max_depth": 4
  },
  "quality": {
    "structure": {
      "enabled": true,
      "warn_root_code_files": 96,
      "max_root_code_files": 120,
      "warn_code_files_per_dir": 12,
      "max_code_files_per_dir": 16,
      "warn_code_dir_depth": 3,
      "max_code_dir_depth": 4,
      "allowed_root_code_files": ["__init__.py", "registry.py"],
      "enforce_role_imports": true
    }
  }
}
```

`runtime.async_max_workers` 是每个 `PipelineRuntime` 自有线程池的最大并发数，必须是正整数，缺省为 4；嵌套 Runtime 继承该数值但仍拥有独立线程池。`runtime.async_flush_timeout` 可以是 `null` 或非负秒数。`runtime.nodeset_max_depth` 必须是正整数，缺省为 4；顶层 pipeline 为 0，普通 nodeset 或 loop body 首次进入为 1，循环迭代不累计。所有已加载及 planned 定义都会检查。程序化显式运行参数优先于 root 配置；现有 `--async-flush-timeout` / runtime profile 仍只覆盖相关运行字段，不提供深度 CLI 参数。

`registry` 相对所属 root 目录解析。`registry.py` 中可选的 `build_base_lib_registry()` / `build_plugin_registry()` 负责声明该 root 下可用资源；具体 pipeline config 继续通过 `--config project/configs/main.jsonc` 指定，并用 id 声明实际使用的资源：

```jsonc
{
  "base_lib": {"modules": [{"id": "math_tools"}]},
  "plugins": [{"id": "project_policy"}],
  "pipeline": {"nodes": [], "edges": []}
}
```

pipeline config 可以用 `nodeset_imports` 导入同 root 或其他 root 的 nodeset：

```jsonc
{
  "nodeset_imports": [
    {"path": "nodesets/task_flow.jsonc"},
    {"root": "vibetrain", "path": "configs/nodesets/train_step.jsonc"}
  ]
}
```

workspace 模式下不要在 pipeline config 顶层声明 `policy`；全局 policy 放根目录 `vibeflow_config.jsonc`。base_lib/plugin 的可用清单放 `project/registry.py`，实际使用清单放 workflow config。审查图只画 workflow config 实际引用的资源。

## verify-kernel

```powershell
python run.py verify-kernel
```

只做完整性检查，不加载 config。失败时会列出：

- `Changed`：manifest 中记录的文件内容变了。
- `Missing`：manifest 中记录的文件不存在。
- `Unexpected`：受保护区域出现了 manifest 未记录的文件，例如把内核源码解包到 `kernel/vibeflow/`。

## validate

```powershell
python run.py validate --config project/configs/main.jsonc
```

输出健康报告。状态可能是：

- `PASS`：通过。
- `CONCERNS`：有 warning，但可运行。
- `FAIL`：配置、拓扑或健康检查失败，不应运行。
- `ERROR`：读取、语法、插件加载等错误。

阅读 warning/error 时不要只看 rule id。先看 `object_type`、`object_id`、`source_location`，再看 `details`。`details.owner` 会标明问题属于顶层 `pipeline` 还是某个 `nodeset:<name>`；flow/data 问题会列出相关 incoming/outgoing edge、direct sources、provider type 或 downstream requirement 摘要；mainline 问题会列出 `source` / `target`、`branch_nodes`、`branch_edges`、`mainline_path` 和 `suggested_fixes`；duplicate warning 会列出具体重复对象和 fingerprint。

当 nested nodeset / loop body 的静态展开让同一个根因被重复发现时，报告会按根因聚合。聚合 finding 的 `details` 会包含 `aggregated: true`、`occurrences` 和 `suppressed_duplicates`。优先修代表 finding 指向的 `owner`、`node`、`required_type`、`direct_sources`；`occurrences` 只是被压缩的重复展开次数，不代表独立 bug 数。

## run

```powershell
python run.py run --config project/configs/main.jsonc --run-root runs
python run.py run --config project/configs/main.jsonc --input input.json --run-root runs
```

运行前会强制健康检查。失败时拒绝运行。

成功或失败都会在运行目录生成报告：

```text
runs/<run_id>/
  input_summary.json
  effective_policy.json
  compiled_graph.json
  health_report.json
  graph.mmd
  graph.txt
  graph.svg
  graph.svg.error.txt      # 仅 SVG 渲染失败时出现
  graph.expanded.svg
  graph.expanded.svg.error.txt  # 仅详细审查 SVG 渲染失败时出现
  runtime_trace.jsonl
  output_summary.json
```

`input_summary.json`、`output_summary.json` 和 `runtime_trace.jsonl` 只写 key、类型、大小、事件和失败摘要，不保存真实 Python 对象内容。训练类对象可以在 node 间按引用传递，但不会被报告序列化。摘要中的 `"scalar": true` 只表示原值是标量，不是业务布尔值 `True`，也无法区分原值是 `True` 还是 `False`。不要从 `output_summary.json` 取业务返回值，更不要对其中的摘要字典做 `bool(...)`。完整 trace 事件只以 JSONL 形式流式写入 `runtime_trace.jsonl`；`RunResult.runtime.*` 和 runtime hook 收到的 `trace` 只包含 summary、`event_count`、`trace_path` 和 `events_streamed=true`，不包含 `events` 列表。

异步 node 会在 trace 中记录 `async_result` / `async_result_join` 或 `async_detached` / `async_detached_done` / `async_detached_failed` 事件。

`--input` 指向一个 JSON 文件，内容会作为初始 context。适合传入字符串、数字、列表、对象等 JSON 数据。若需要传递模型、tensor、数据库连接等真实 Python 对象，应写自定义启动器调用 `run_checked(..., initial=...)`，不要把这些对象写进 JSON。

运行选项：

```powershell
python run.py run --config project/configs/main.jsonc --runtime-profile debug
python run.py run --config project/configs/main.jsonc --runtime-profile train
python run.py run --config project/configs/main.jsonc --trace boundary --execution compiled --no-node-hooks
python run.py run --config project/configs/main.jsonc --allow-planned-stub
```

- `--runtime-profile debug`：完整 trace，所有 hook 开启，执行模式为 `plan`。
- `--runtime-profile train`：边界 trace，run/block hooks 开启，node/nodeset hooks 关闭，执行模式为 `compiled`。
- `--trace full|boundary|off`：控制 trace 详细程度；`full` 的完整事件无界写入 `runtime_trace.jsonl`，不会保存在内存事件列表中。
- `--execution plan|block|compiled`：控制执行模式；`compiled` 会优先使用 generated graph/nodeset/loop block，不适合的区域回退 plan；`block` 是严格模式，不能 block 化会启动前报错。
- `--async-flush-timeout`：run 结束时等待 detached async task 的秒数。
- `--allow-planned-stub`：仅开发测试用。开启后只有 `planned_behavior.kind == "python_stub"` 的 planned node/nodeset 可执行；`blocking` 和 `transparent` 仍拒绝运行。

## mermaid / ascii / svg

```powershell
python run.py mermaid --config project/configs/main.jsonc --output reports/graph.mmd
python run.py mermaid --config project/configs/main.jsonc --expand-nodesets --output reports/graph.expanded.mmd

python run.py ascii --config project/configs/main.jsonc --output reports/graph.txt
python run.py ascii --config project/configs/main.jsonc --expand-nodesets --output reports/graph.expanded.txt

python run.py svg --config project/configs/main.jsonc --output reports/graph.svg
python run.py svg --config project/configs/main.jsonc --expand-nodesets --output reports/graph.expanded.svg
python run.py svg --config project/configs/main.jsonc --expand-nodesets --mermaid-max-text-size 800000 --output reports/graph.expanded.svg
python run.py svg --config project/configs/main.jsonc --expand-nodesets --review-fragment-max-width 6400 --output reports/graph.expanded.svg
```

- Mermaid 图用于审查源 flowchart。
- ASCII 图不依赖外部渲染器，适合终端快速查看。
- SVG 图使用 Mermaid CLI 渲染，最适合人工审查标准流程图形状。首次使用前需要在 `kernel/tools/mermaid-renderer/` 执行 `npm install`。不要求系统预装 Google Chrome；VibeFlow 会优先使用 Puppeteer 自己安装/缓存的浏览器，其次尝试非 snap 的系统 Chrome/Chromium。`/snap/bin/chromium` 会被跳过，因为它在 Puppeteer/mermaid-cli 下常见 profile lock 启动失败。
- SVG 渲染默认传入 Mermaid 配置：普通图 `maxTextSize=200000`、`maxEdges=2000`；`--expand-nodesets` 图 `maxTextSize=500000`、`maxEdges=5000`，并固定采用 `review-columns` SVG composer，把主流程、当前 workflow 实际启用的 plugins/base_lib 和展开 nodeset 分列展示。SVG 保持 `htmlLabels=false`，但内核会对原生 SVG 文本做标题加粗、字段名前缀加粗和字段行左对齐增强。如果展开图仍超过 Mermaid 限制，可用 `--mermaid-max-text-size` 和 `--mermaid-max-edges` 覆盖；如果希望审查图单个片段显示得更宽，可用 `--review-fragment-max-width` 调整默认 `3200px` 上限。
- `graph.expanded.mmd` 是展开 Mermaid 源码调试产物，不是详细审查 SVG 的输入。不要把它直接交给 Mermaid CLI/mmdc 转成 SVG；详细审查 SVG 必须由 `python run.py svg --expand-nodesets --output reports/graph.expanded.svg` 生成，才能使用 review-columns/detail-panel 布局。

图形输出会显示：

- 只显示显式 `pipeline.edges`；`requires/provides` 不会自动画理论数据边。
- workspace 模式下 node、nodeset、base_lib 和 plugin 的 root/source。
- `info.available_resources`：root registry 中可用的 base_lib/plugin 清单。
- `info.resources` / `info.effective_resources`：当前 workflow config 实际引用并生效的 base_lib/plugin 清单；Mermaid/MMD/SVG 使用这一份。
- 同步主线 edge 加粗，显式 data bypass edge 虚线，async 相关 edge 保持 async 语义。
- 标准 flow_kind 形状。
- node / nodeset 契约 key。
- 中文说明和元数据。
- `when` 条件。
- planned / external 标记。
- health finding 标记。

## quality

```powershell
python run.py quality --path project
python run.py quality
```

这是通用代码质量检查工具，可检查不基于内核的普通 Python 项目。显式 `--path` 保持单路径扫描；未传 `--path` 且存在 workspace 配置时，会扫描所有 `quality_enabled` roots。它会检查文件长度、函数复杂度、依赖链长度、重复 AST 形状等。

workspace 默认还会读取每个 root 的 `quality.structure` 做目录治理：root 总代码文件数默认 warning/error 为 96/120；单个代码目录内直接 `.py` 数默认 warning/error 为 12/16；目录深度默认 warning/error 为 3/4。这个规则的目标是让 `nodes/`、`base_lib/`、`plugins/` 按功能继续拆子目录，而不是单纯压低 root 总文件数。显式 `--path` 扫描普通目录时默认不启用这些 root 结构限制；需要时加 `--enable-structure-limits`。

## 自定义启动器

如果模板 `run.py` 不够用，可以在项目根目录写自己的启动器，但仍应复用 workspace 入口：

```python
from pathlib import Path
import sys

ROOT = Path(__file__).resolve().parent
sys.path.insert(0, str(ROOT / "kernel" / "vibeflow-kernel.zip"))
from vibeflow import RuntimeOptions, load_workspace_config, run_workspace_checked

workspace = load_workspace_config(ROOT / "vibeflow_config.jsonc")
result = run_workspace_checked(
    "project/configs/main.jsonc",
    workspace=workspace,
    initial={"value.in": 1},
    run_root="runs",
    runtime_options=RuntimeOptions(trace="boundary", execution="plan"),
)
value = result.context.get("value.out")["value"]
print(result.health.status, result.run_dir, value)
```

成功运行后，`run_workspace_checked` / `run_checked` 返回的 `CheckedRunResult.context` 保留本次运行的真实 envelope。自定义 adapter 或启动器应通过 `result.context.get("value.out")["value"]` 读取业务值；`output_summary.json` 只用于审计结构，不是可恢复的结果存储。

自定义启动器也应保留 kernel manifest 校验逻辑；除非你非常清楚后果，不要跳过模板 `run.py` 的完整性检查。
